# 即影 · 商业化产品需求文档（PRD）

> 版本：v1.0
> 基于现有代码能力 + 商业逻辑梳理
> 每个功能项标记技术是否已具备

## 一、产品定位

**即影 = AI 自媒体内容创作平台**
帮助用户从 0 到 1 运营自媒体账号：AI 分析赛道 → AI 生成内容 → 用户审核 → 一键发布

## 二、会员体系与权限

### 2.1 用户三种状态及各状态行为

| 状态 | 定义 | 点击一级导航行为 |
|------|------|------------------|
| 🚶 **游客** | 未登录/无 session | 看到所有导航名称，点任何功能 → 跳转 `20元开启你的自媒体公司` 广告页 |
| 👤 **注册会员** | 已登录但未付费 | 看到所有导航名称（AI引擎展示15个名字但锁定），点任何功能 → 跳转 `20元` 广告页 |
| 💎 **付费会员** | 已完成 ¥20 支付 | 全部功能可用。其中每日内容引擎要求额外完成「绑定/注册自媒体账号」后才可完整使用 |

### 2.2 付费会员分级

```
付费会员

  └ 未绑定自媒体账号 → 可使用影片工厂/数字人/图片社，每日内容提示"请先绑定账号"
```

### 4.1 🤖 AI引擎（原Agent智能调度中心）

**用户故事：** 用户点击后看到 15 个 AI 引擎的名称列表。

- 未登录/未付费：只显示引擎名称和图标，点击弹出"开通会员"提示
- 已付费：点每个引擎 **直接进入对应自动化工作流**

**核心概念：每一个 AGENT = 一条自动的工作流。** 用户点击 agent 不是看说明文档，是直接干活。每个 agent 打开后直接展示该工作流的输入界面。

**关键引擎——拆解专家（示例工作流）：**
```
用户点击"拆解专家"
       ↓
直接弹出：竞品链接输入框
       ↓
用户粘贴爆款视频链接
       ↓
调 DeepSeek 拆解（钩子/节奏/结构/情绪）
       ↓
返回结构化拆解报告
```

| 技术点 | 状态 |
|--------|------|
| 15 个 Agent 展示 | ✅ 已有 15 个 agent 页面 |
| 付费锁定 | ❌ 需实现 |
| 拆解专家（链接输入+DeepSeek拆解） | ✅ `/api/trends/deconstruct` + `ReferenceUrlInput` 已有 |

### 4.2 📋 每日内容

**用户故事：** 付费会员进入后，看到自己选的内容赛道和已绑定的自媒体平台。下方是竞品链接输入框和爆款搜索按钮。

**关键流程：**
```
用户已选赛道="程序开发" + 已绑定小红书
         ↓
页面加载：赛道=程序开发（不可改）、平台=小红书（不可改）
         ↓
输入竞品链接（可以是抖音链接，不限于本平台）
         ↓
点击"搜索抖音爆款结构"（按钮平台名=输入链接的平台）
         ↓
显示 Top 3 爆款 → 选一个 → DeepSeek 拆解
         ↓
生成 3 篇文案 → 选一篇 → 配图 → 下载
```

| 技术点 | 状态 |
|--------|------|
| 从 launch 页读取赛道+平台 | ⚠️ 已有但数据传输不稳定 |
| 竞品链接输入（不限平台） | ✅ `ReferenceUrlInput` 已支持 |
| 搜索按钮平台自动匹配 | ⚠️ 需要改：从链接自动识别平台 |

### 4.3 🎬 即刻影片工厂

#### 4.3.1 微短剧

**【重要】链接性质区分：**
- 每日内容的链接 = **竞品链接**（用于分析爆款结构，参考其营销套路）
- 微短剧的链接 = **参考链接**（用于提取剧情结构+视觉风格，作为创作参照）
- **这两个不能混淆。** 微短剧粘贴的必须是短剧/影视类链接，系统参照其剧情结构和风格生成用户自己的短剧。

**用户故事：** 用户粘贴一个微短剧的参考链接 + 输入自己的创意 → 系统参照链接的结构和风格 + 结合用户创意 → 生成类似结构但不同剧情的短剧。

**角色驱动的核心逻辑：**
```
用户选择/创建角色模板（CharacterCreator）
  → 上传角色正面/侧面/背面参考图
  → 填写角色外貌、服饰、性格
       ↓
角色参考图 = 图生图的底图（类似 img2img）
  → 所有故事创作围绕此角色展开
  → 所有分镜设计保持角色一致
  → 所有视频合成以角色参考图为锚点
  → ⚠️ 短剧全部围绕该角色，不是"参考图辅助"而是"以图为纲"
```

**电影制作级参数设定（必须可配置）：**

| 参数 | 说明 | 举例 |
|------|------|------|
| **视觉风格** | 整体画面风格/镜头语言 | FPV无人机视角 / 电影质感 / 赛博朋克 / 急速感 / 广角镜头 / 色彩鲜艳 |
| **核心主体** | 视频中的角色 | 用户创建的角色模板（必选） |
| **初始环境** | 故事发生地+氛围 | 拥挤的摩洛哥集市，阳光强烈 |
| **时间轴** | 时间背景 | 正午 / 黄昏 / 夜晚 / 黎明 |
| **运镜方式** | 镜头运动 | 跟随 / 环绕 / 第一人称 / 航拍 / 推拉摇移 |
| **动作说明** | 具体场景动作描述 | 镜头紧跟主角穿过拥挤的人群，推翻水果摊 |

> 需要还原电影制作的各种拍摄场景参数，才能产出有画面感的高质量短剧。

**技术架构：**
```
竞品链接 → /api/trends/deconstruct → 拆解结构
         ↓
用户角色模板 → character-engine → 角色参考图
         ↓
用户参数（风格/环境/运镜等）→ Prompt Engine
         ↓
合并 → executeStage("story_genesis") → 剧本
     → executeStage("script_breakdown") → 分镜
     → visual_generation（参考图+上帧→Seedance）
     → StoryboardVideoRenderer → MP4
```

| 技术点 | 状态 |
|--------|------|
| 竞品链接拆解 | ✅ 已有 |
| 角色模板+参考图 | ✅ 已有 |
| 6阶段流水线 | ✅ 已有 |
| 视频参数（风格/运镜/环境等） | ❌ 需新增 UI 和 prompt 注入 |
| 全自动运行 | ✅ 已有 |

#### 4.3.2 漫剧

**与微短剧同架构，使用完全相同的电影制作级参数系统（镜头焦段/景别/拍摄角度/运镜方式/光线设计/色彩基调/环境设定/时间背景/氛围情绪/动作说明/声音设计/剪辑节奏）。** 差异点：
- 视觉风格默认值不同：默认为「日系动漫风格」（可选：国风水墨/赛博朋克/皮克斯3D/复古漫画/手绘水彩/厚涂油画风等）
- 分镜更强调画面冲击力、色彩张力和视觉风格一致性
- 同样需要角色驱动（用户创建的角色模板），角色在漫剧中需保持二次元化风格一致
- 动作说明同样是自由文本输入，不是写死的选项
- 视频渲染用 StoryboardVideoRenderer

#### 4.3.3 知识图谱

**用户故事：** 用户输入知识点，AI 生成结构化的知识讲解视频（Canvas 动画渲染）。

|--------|------|
| 文案→幻灯片解析 | ✅ `slide-engine.ts` |

> 参考项目：[calesthio/OpenMontage](https://github.com/calesthio/OpenMontage)（AGPL-3.0）

### 10.1 即影 vs OpenMontage 架构对比

| 维度 | OpenMontage | 即影（现状） | 可否借鉴 |
|------|-------------|-------------|----------|
| **编排方式** | Agent-first，无中央编排器，AI助手自行决策 | 固定6阶段流水线（PIPELINE_STAGES硬编码） | ✅ 已有P1-2导演模式，可继续深化 |
| **技能系统** | 3层知识架构：工具→技能→知识包 | 15个独立Agent页面，无统一技能管理层 | ⚠️ 需重构，暂不列为优先级 |
| **渲染引擎** | Remotion（React）+ HyperFrames（HTML/CSS/GSAP）双引擎 | Canvas + MediaRecorder（刚完成） | ✅ 方向一致，即影的Canvas方案已验证可行 |
| **质量门禁** | 生成前校验 + 生成后自检（ffprobe/帧采样/音频分析） | 仅有质检引擎（video-cognition-qa.ts） | ✅ 可参考，需新建 |
| **成本管控** | 预执行估算，默认0上限，/usr/bin/bash.5/次审批 | 无 | ❌ 待新建 |
| **参照视频** | 粘贴YouTube/Reel链接 → 分析结构+风格 → 生成差异化方案 | 已有ReferenceUrlInput | ✅ 方向一致，即影已实现基础版 |
| **Provider评分** | 7维度加权评分选模型（任务匹配/质量/控制/可靠性/成本/延迟/连续性） | 简单任务分类（classifyTask） | ✅ 可参考改进ModelRegistry |
| **免费素材** | Archive.org/NASA/Wikimedia/Pexels/Unsplash/Pixabay | 仅Pexels/Unsplash（刚接入） | ✅ 可扩更多源 |

### 10.2 可直接借鉴的设计理念（不涉及AGPL代码）

#### 10.2.1 导演工作流改良（对应即影的6阶段流水线）

OpenMontage 的 7 阶段流程：
```
research → proposal → script → scene_plan → assets → edit → compose
```

即影当前是 6 阶段线性流水线。可借鉴的点：
- **加「研究」阶段**：在故事创世前，先搜索相关素材/风格/竞品 → 已有 Tavily 搜索能力，但未前置到流水线
- **加「提案」阶段**：AI 先给用户 2-3 个创作方向让用户选，确定方向后再执行后续 → 当前用户一句话直接生成，缺少方向选择
- **场景计划独立于分镜**：场景计划（scene_plan）是"拍什么"，分镜（script_breakdown）是"怎么拍" → 即影合并了，可拆分

#### 10.2.2 Provider 评分系统（对应即影的 ModelRegistry）

参考设计理念（不复制代码）：
- 每次调用模型时，按7维度评分选择最优模型
- 评分维度：任务匹配/输出质量/控制能力/可靠性/成本效率/延迟/连续性
- 影响：视觉生成阶段自动选择最优模型（Seedance/Kling/Runway）
- 工作量：1 天

#### 10.2.3 渲染引擎策略

即影当前只用了 Canvas + MediaRecorder。可扩展：
- **静帧内容**（知识讲解/幻灯片）→ Canvas 动画录制 ✅ 已有
- **运动画面**（微短剧/漫剧）→ StoryboardVideoRenderer ✅ 已有
- **纯文本/图表**（数据报告/产品详情）→ HTML2Canvas → 视频 → ❌ 待建

#### 10.2.4 质量门禁

OpenMontage 有 pre-compose validation + post-render self-review。即影已有 QC 质检引擎但未自动化为流水线中的门禁。
- **Pre-compose**：生成前检查脚本质量是否达标
- **Post-render**：生成后自动分析视频质量
- 工作量：2 天

#### 10.2.5 成本追踪

OpenMontage 有预算控制。即影完全无成本控制。
- 建议加简单的 token 计数器 + 每次生成前估算
- 工作量：1 天

### 10.3 暂不建议采纳的

| 设计 | 原因 |
|------|------|
| Agent-first 无中央编排 | 即影用户需要明确的"一键生成"体验，不需要 agent 自己决策 |
| 500+ agent skill markdown 文件 | 即影是面向非技术用户的商业化产品，不是开发者工具 |
| HyperFrames（HTML/GSAP） | 即影的 Canvas 方案已覆盖相同能力，不需要额外渲染引擎 |
